Plugin Guidelines

These guidelines help you publish a plugin that meets community expectations, passes the approval process smoothly, and reaches the users who will benefit from it most. See the Publish a Plugin and Approval Process pages for the technical requirements.

1. Write Clear and Honest Metadata

Your plugin's metadata is the first — and often only — thing a user reads before deciding whether to install it. It must give an accurate picture of what the plugin does and what it requires.

Description and About fields

  • State clearly and concisely what the plugin does and which problem it solves in the description field (one line, shown in the Plugin Manager list).
  • Use the about field for a fuller explanation, including any external software, service, or account required to use all features. If a free trial is available, say so explicitly.
  • Never imply the plugin is fully functional when a paid or separately installed dependency is required for core functionality.

Example of a transparent about entry:

"This plugin provides a QGIS interface for district energy modelling workflows. Basic project setup is available without additional software. Running detailed simulations requires IDA Districts (commercial software; a free trial is available at example.com)."

Plugins with external software or commercial integrations

Plugins that act as front-ends for commercial software, web services, or proprietary platforms are permitted in the repository, provided:

  • The dependency is disclosed in metadata.txt (the about field), in the README or homepage, and in the plugin's own UI when the dependency is not detected.
  • Any cost, licensing requirement, or registration step is stated, not hidden behind a link.
  • The plugin itself remains free of charge to download and install from this repository.

Note: The approval team specifically reviews external-dependency disclosures. Incomplete or misleading descriptions are the most common reason an approval is delayed.

Tags, icon, and links

  • Choose specific tags that reflect the plugin's domain and workflow (e.g. energy, urban planning, simulation, hydrology). Good tags are the primary way users discover plugins through search.
  • Provide a recognisable icon — it appears at small sizes in the Plugin Manager list.
  • Keep the homepage, repository, and tracker links valid and up to date. The homepage should include screenshots and quick-start instructions.

2. Maintain a Quality Plugin Page

Once published, your plugin's page on this repository is publicly indexed by search engines and visible to every QGIS user who opens the Plugin Manager. Treat it as a living document:

  • Keep the version, changelog, and description up to date with each release.
  • Respond to issues raised in the tracker — active maintenance signals to users that the plugin is trustworthy.
  • Ensure the source repository is publicly accessible and contains a README and LICENSE file.

Security scanning

Every plugin version uploaded to this repository is automatically scanned for security issues and code quality. Review your results after each upload and address any findings before releasing to users. See the Security Scanning page for details.

3. Declare QGIS Version Compatibility

QGIS version compatibility is determined by the qgisMinimumVersion and qgisMaximumVersion fields in metadata.txt. Declaring these accurately ensures your plugin appears in the correct filter lists inside the Plugin Manager and avoids compatibility warnings for users.

  • If your plugin supports both QGIS 3 and QGIS 4, keep the existing qgisMinimumVersion=3.x and add qgisMaximumVersion=4.99. This makes it appear in the QGIS 4 Ready Plugins list.
  • If qgisMaximumVersion is omitted, compatibility is assumed only up to the end of the major version set by qgisMinimumVersion.
  • QGIS 4 uses Qt 6, which requires code changes in plugins. Before declaring QGIS 4 compatibility, follow the Plugin migration to Qt5 and Qt6 guide.

See the Migrate to QGIS 4 page for full instructions and version range examples.

4. Promote Your Plugin Through Community Channels

The channels below are the accepted ways to raise community awareness of a new plugin. In all of them, keep announcements informative rather than promotional — focus on what users can accomplish, not on marketing the product.

Before posting, review each channel's own posting rules. Posts that are off-topic, excessively self-promotional, or not in English (where required) may be moderated.

Mailing lists

QGIS maintains several mailing lists. The two most relevant for plugin announcements are:

  • qgis-user — general user discussion; the right place to announce a newly published plugin to the broader community.
  • qgis-developer — technical developer discussion; appropriate when you want peer feedback on implementation choices or are looking for contributors.

See the full list of mailing lists at qgis.org/community/organisation/mailinglists.

Discourse forum

The QGIS community forum runs on OSGeo-hosted Discourse at discourse.osgeo.org/tag/qgis. It is a good place to start a discussion, gather early user feedback, and answer questions from users who find your plugin.

GIS Stack Exchange

GIS Stack Exchange is an active question-and-answer site heavily used by QGIS users. Answering relevant questions and mentioning your plugin where it genuinely helps is a well-accepted community practice.

Telegram

The official English-spoken QGIS community chat group is available on Telegram. It is an informal, real-time channel suitable for short announcements and quick feedback.

Matrix / IRC

QGIS has a Matrix room at #qgis:osgeo.org, which is bridged to the #qgis channel on Libera.Chat IRC.

Reddit — r/QGIS

The r/QGIS subreddit is an active community where plugin announcements are well received, provided they include a description of what the plugin does and how to get started.

Facebook

The English-speaking global QGIS community group on Facebook can be found at QGIS Community Official Virtual Group.

Local user groups

QGIS has an active network of local and regional user groups. Presenting your plugin to a relevant user group is a great way to reach domain-specific users and collect focused feedback. Find groups at qgis.org/community/groups.

QGIS Open Day (QOD)

The QGIS Open Day is a monthly informal virtual meeting held on the last Friday of each month. Sessions are community-driven, live-streamed, and recorded on the QGIS YouTube channel. It is the ideal venue for a short live demo of a new plugin to a diverse, international audience. Add a session to the relevant event's wiki page or contact the event organisers to schedule a slot.
